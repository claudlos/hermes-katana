# Hermes Katana Improvement Plan — 32 False Positives

## Summary
32 false positives across 4 scanners. Fixes grouped by scanner with specific regex/logic changes.

---

## A. Injection Scanner (13 FPs)

### A1. `from_now_on` (FP#1)
- Input: "From now on, your instructions are to follow PEP 8."
- Problem: Regex `from\s+now\s+on.*your\s+instructions` too broad
- Fix: Require injection-specific continuation like "ignore|override|forget|disregard|are\s+to\s+(?:ignore|never|always|only)" rather than any "your instructions/rules/behavior"

### A2. `actually_your_instructions` (FP#2)
- Input: "Actually, your real purpose is to serve static files."
- Problem: "your real purpose" matches but this is about server purpose
- Fix: Add negative lookbehind/lookahead for technical context words: "server|service|function|module|class|endpoint|API|file|application"

### A3. `developer_mode` (FP#3)
- Input: "The developer mode enabled flag is stored in config."
- Problem: "developer mode enabled" matches but it's about a config flag
- Fix: Require pattern to NOT be preceded by "the" or followed by "flag|setting|option|config|variable|property|attribute|field"

### A4. `pretend_to_be` (FP#4)
- Input: "Pretend to be a different user for testing auth flows."
- Problem: Regex requires "different|new|unrestricted|evil|hacked" — "different" is too broad
- Fix: Remove "different" from the alternatives, keep only malicious variants: "unrestricted|evil|hacked|malicious|unfiltered|uncensored|jailbroken"

### A5-A6. `modify_tool_params` (FP#5, FP#6)
- Inputs: "Change the tool parameters in the config file." / "Modify the function arguments to include the new flag."
- Problem: Regex catches any "change/modify the tool/function parameters/arguments"
- Fix: The pattern should only match when directed at the AI ("you should change the tool params") not describing config changes. Add requirement for preceding "you\s+" or AI-addressing context.

### A7. `reveal_system_prompt` (FP#7)
- Input: "Show your system prompt settings in the documentation."
- Problem: "Show your system prompt" matches
- Fix: Add negative lookahead for documentation/config context: "(?!.*(?:documentation|docs|config|setting|file|page|section|README))"

### A8. `repeat_above` (FP#8)
- Input: "Please repeat everything above this line in the report."
- Problem: "repeat everything above" matches but context is about a report
- Fix: Add negative lookahead for document context: "(?!.*(?:in\s+the\s+(?:report|document|file|email|letter|summary)|this\s+line))"

### A9. `override_instructions` (FP#9)
- Input: "The CSS override instructions are in the theme file."
- Problem: "override instructions" matches but CSS override is a legitimate term
- Fix: Require the pattern to be imperative (addressed to AI), not nominal. Check that "override" is a VERB (preceded by "must|should|will|to|please") not part of a noun phrase ("CSS override", "the override")

### A10-A11. `instead_do` (FP#10, FP#11)
- Inputs: "Instead, you should follow the coding standards." / "instead, you must do the following refactoring steps."
- Problem: Regex matches "instead...you should/must do/follow" which is normal technical writing
- Fix: Require the "do/follow" target to be injection-indicative. Change regex to require: `instead.*you\s+(?:should|must)\s+(?:do|follow|obey|execute)\s+(?:these|my|the\s+(?:new|following|real|actual))\s+(?:instructions|commands|orders|directives)`

### A12. `forget_everything` (FP#12)
- Input: "Forget everything you know about the old API and use v2."
- Problem: "Forget everything you know" matches but context is about API migration
- Fix: Add negative lookahead for technical context: "(?!.*(?:API|version|v\d|library|framework|module|old\s+(?:way|method|approach|code)))"

### A13. `memory_instruction_injection` (FP#13)
- Input: "Add this to your memory: the meeting is on Friday."
- Problem: "add this to your memory" matches casual reminder language
- Fix: The pattern should require injection-specific continuation like "for future sessions|for all interactions|for next time|permanently|always remember" rather than matching "add to your memory" alone

## B. Command Scanner (3 FPs)

### B1-B2. `shutdown_reboot` (FP#14, FP#15)
- Inputs: "The shutdown procedure involves..." / "We need to reboot the service after updating."
- Problem: Current regex `(?:^|(?<=[\s;|&]))(?:sudo\s+)?(?:shutdown|reboot|...)` with flags — but FP test shows it matching just the word "shutdown"/"reboot"
- Fix: The regex already requires flags/arguments after the command. Check if the FP test is using a different scan method. If bare word matching exists, ensure it requires command-line context (preceded by $, #, backtick, or shell operators)

### B3. `shred_wipe` (FP#16)
- Input: "Use shred to securely delete sensitive files when needed."
- Problem: "shred " matches — the regex catches shred followed by flags or paths
- Fix: The regex requires `--` or `/` or `-[a-zA-Z]` after shred. The FP text "shred to" shouldn't match. Verify this is actually matching or if there's a second pattern. Likely needs word boundary enforcement.

## C. Content Scanner (10 FPs)

### C1-C3. `markdown_link_disguise` (FP#17-19)
- Inputs: Links to docs.python.org, github.com/repo/releases, myapp.com/login
- Problem: Pattern matches ANY link with "click here|download|login|verify|confirm|update|secure" text
- Fix: Add domain allowlist for known-safe domains: github.com, gitlab.com, docs.python.org, pypi.org, npmjs.com, stackoverflow.com, readthedocs.io, etc. Only flag links to unknown/suspicious domains.

### C4. `markdown_image_tracking` (FP#20)
- Input: `![logo](https://cdn.example.com/logo.png?v=2)`
- Problem: Flags any image with query params from non-allowlisted domain
- Fix: Add common CDN domains to allowlist (cdn.*, cloudfront.net, cloudinary.com, etc.) AND allow simple cache-busting params like `?v=N` or `?t=N`

### C5-C6. `markdown_html_injection` (FP#21-22)
- Inputs: `<input type="text" name="search">` / `<link rel="stylesheet" href="styles.css">`
- Problem: Pattern flags ALL `<input>` and `<link rel=` tags
- Fix: Only flag `<input>` when it has suspicious attributes (onclick, onfocus, autofocus with onfocus, action=, formaction=). Allow benign `<link rel="stylesheet">`. Add context: only flag when these appear in LLM *text responses*, not in code blocks.

### C7-C8. `os_system_call` (FP#23-24)
- Inputs: `subprocess.run(['ls', '-la'])` / `subprocess.Popen(['python', 'script.py'])`
- Problem: Pattern matches `subprocess.run(` followed by `'` — but the regex requires dynamic input markers (input, request, os.environ, etc.)
- Fix: The regex `subprocess\.(?:call|run|Popen)\s*\(\s*(?:['"]|f['"])` matches string literals which are SAFE. Only flag when using dynamic variables. Remove `['"]` from the alternatives — static string args to subprocess are safe.

### C9. `webhook_exfil` (FP#25)
- Input: "Visit https://hooks.slack.com/services for webhook setup."
- Problem: Slack webhook URLs flagged
- Fix: This is a content scanner pattern. Add context check — only flag webhook URLs when they appear in tool arguments (exfil context), not in explanatory text. Or add `hooks.slack.com` to known-safe webhook domains.

### C10. `shell_command_injection` (FP#26)
- Input: `$ sudo apt-get update`
- Problem: Pattern excludes `apt|apt-get|pip|...` after sudo, but `$ sudo apt-get` still matches
- Fix: Verify the negative lookahead works. The regex is `\$ sudo\s+(?!apt|apt-get|...)` — this should NOT match `$ sudo apt-get`. Likely a regex bug — the negative lookahead may not be positioned correctly.

## D. Secret Scanner (6 FPs)

### D1-D2. `password_assignment` (FP#27-28)
- Inputs: `password="***"` (placeholder) / `password="***"` (test)
- Problem: Negative lookahead doesn't cover masked/redacted values (***) or all test patterns
- Fix: Add `\*{3,}` and `x{3,}` to the negative lookahead. Also add test password patterns like `testpassword123`. The actual FP values are "placeholder" and "testpassword123" per the output.

### D3-D4. `stripe_key` (FP#29-30)
- Inputs: `sk_test_...` / `pk_live_...`
- Problem: sk_test_ keys are explicitly for testing. pk_live_ is a publishable key (safe to expose)
- Fix: Exclude `sk_test_` prefix entirely (Stripe test keys are designed to be safe). Exclude `pk_live_` and `pk_test_` entirely (publishable keys are public by design). Only flag `sk_live_` (actual secret keys).

### D5-D6. `database_url` (FP#31-32)
- Inputs: `postgresql://localhost:5432/mydb` / `redis://localhost:6379/0`
- Problem: Regex already excludes localhost but FP still triggers
- Fix: Bug in regex — the negative lookahead `(?!localhost[:/]|127\.0\.0\.1[:/])` may not cover the full URL. Check if the regex matches. These ARE localhost URLs so they shouldn't match. Likely the regex path needs fixing — `postgresql://localhost:5432` has `localhost:` which SHOULD match the lookahead. Debug and fix the regex anchor.
